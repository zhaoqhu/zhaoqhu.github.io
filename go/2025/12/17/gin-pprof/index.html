指标分类与核心说明

Go net/http/pprof 暴露的10个性能分析端点，按分析对象分类如下，涵盖内存、CPU、阻塞、并发、线程、执行流程等维度，配合 go tool 可精准定位性能瓶颈。

指标详情表格

分类 指标名称 核心用途 go tool 分析命令/步骤 关键参数/注意事项
内存分析 allocs 累计内存分配（含已释放对象），记录程序启动以来的所有内存分配历史（字节/对象数）。 go tool pprof http://localhost:8080/debug/pprof/allocs - 默认分析 alloc_space（累计分配）
- 用 top 看累计分配大户
- 结合 heap 对比当前活跃内存
  heap 堆内存快照（当前活跃对象，未被GC回收的内存），记录实时内存占用。 go tool pprof http://localhost:8080/debug/pprof/heap - 默认 inuse_space（当前活跃），用 sample_index=alloc_space 切换累计分配
- 多次采样后用 diff 对比泄漏
- 用 list 定位代码行
CPU分析 profile CPU耗时采样（默认10秒），记录函数CPU占用时长（纳秒级）。 go tool pprof http://localhost:8080/debug/pprof/profile?seconds=30 - seconds=N 指定采样时长（建议30秒）
- 用 top 找CPU热点，list 定位代码行
- 用 web 可视化调用图
阻塞分析 block 阻塞事件采样（channel操作、IO等待等），记录goroutine阻塞时长/次数。 go tool pprof http://localhost:8080/debug/pprof/block?seconds=30 - 需先调用 runtime.SetBlockProfileRate(1) 启用采样
- 用 top 看阻塞热点，list 定位阻塞代码
  mutex 锁竞争事件采样（sync.Mutex），记录goroutine因锁竞争的阻塞时长/次数。 go tool pprof http://localhost:8080/debug/pprof/mutex?seconds=30 - 需先调用 runtime.SetMutexProfileFraction(1) 启用采样
- 用 top 看锁竞争热点
并发分析 goroutine 当前所有goroutine的堆栈跟踪（含状态：运行中、阻塞、死锁）。 go tool pprof http://localhost:8080/debug/pprof/goroutine - 用 curl http://.../goroutine?debug=2 > goroutines.txt 导出堆栈
- 搜索 chan receive/lock 定位阻塞/泄漏
线程分析 threadcreate 线程创建事件采样，记录新线程创建的堆栈（含创建耗时）。 go tool pprof http://localhost:8080/debug/pprof/threadcreate?seconds=30 - 用 top 看线程创建热点
- 适合分析线程创建开销（如频繁创建线程）
执行流程分析 trace 执行时序跟踪（goroutine调度、GC、系统调用、请求处理流程），生成时序日志。 1. 下载trace文件：curl -o trace.bin 'http://.../trace?seconds=5'
2. 分析：go tool trace trace.bin		 - 需先下载二进制文件（trace.bin），再用 go tool trace 打开
- 可视化界面看甘特图、goroutine分析
配置查看 cmdline 程序启动命令行参数（如 ./main -port=8080）。 go tool pprof http://localhost:8080/debug/pprof/cmdline - 直接返回文本，查看启动配置（环境变量、GOMAXPROCS等）
调试工具 symbol 符号表查询接口，将内存地址转换为函数名（供调试工具使用）。 go tool pprof http://localhost:8080/debug/pprof/symbol - 日常分析极少用，仅调试时解析崩溃堆栈中的未知地址

关键说明

1. 分类逻辑

按分析对象划分（内存、CPU、阻塞等），便于快速匹配问题类型：

  • 内存问题 → heap（当前泄漏）+ allocs（累计分配）
  • CPU问题 → profile（CPU热点）
  • 阻塞/并发问题 → goroutine（状态）+ block（阻塞事件）+ mutex（锁竞争）
  • 时序问题 → trace（甘特图分析延迟）

2. 通用命令与避坑点

场景 核心命令 避坑点
内存泄漏定位 go tool pprof http://.../heap → 多次采样 → diff 对比 → list 定位代码行 默认 inuse_space 仅看活跃内存，需切 alloc_space 看累计分配；小泄漏需多次采样
CPU热点定位 go tool pprof http://.../profile?seconds=30 → top → list → web 采样时长建议30秒（覆盖多轮请求），避免短时波动干扰
阻塞事件分析 先 runtime.SetBlockProfileRate(1) → go tool pprof http://.../block?seconds=30 未启用采样时无数据，需代码中显式开启
trace时序分析 curl -o trace.bin 'http://.../trace?seconds=5' → go tool trace trace.bin 需下载二进制文件（非URL直接分析），用可视化界面看甘特图

3. 核心场景速查

  • “内存涨了但heap没反应” → 用 alloc_space 维度 + 多次采样 diff 对比
  • “CPU占用高但不知道哪里慢” → profile 采样 + top 找热点函数
  • “goroutine数量爆炸” → goroutine 导出堆栈 + 搜索 chan receive/lock 定位泄漏源
  • “请求延迟忽高忽低” → trace 甘特图看GC/系统调用与请求的时序重叠
